## Вариант 1: WordPress + Elementor Pro

### Суть подхода

Визуальный drag-and-drop конструктор страниц поверх WordPress. Самый популярный page builder в мире (16+ млн сайтов). Всё строится мышкой: перетащил виджет, настроил стиль, опубликовал. ACF Pro для кастомных полей кухонь, RankMath для SEO.

### Плюсы

1. **Самый низкий порог входа.** Контент-менеджер без опыта разработки сможет менять тексты, фото, цены через визуальный редактор за 1-2 дня обучения. Для 42 карточек кухонь это критично.
2. **Огромная экосистема.** 100+ готовых аддонов (JetEngine для фильтрации каталога, JetSmartFilters для AJAX-фильтров, Dynamic.ooo для динамического контента). Квиз-калькулятор реализуется через плагин (Jetsteps или Elementor Form Steps).
3. **Быстрая разработка каталога.** Theme Builder + Loop Builder позволяют создать шаблон карточки кухни за 2-3 часа и автоматически применить ко всем 42 моделям. Архив CPT Kitchen с фильтрацией по типу/стилю/материалу — из коробки через JetEngine.
4. **Встроенные формы с webhook.** Elementor Pro Forms поддерживает Actions After Submit → Webhook. Интеграция с n8n: форма -> webhook -> n8n -> Telegram/Google Sheets/BlueSales. Без единой строки кода.
5. **Максимальная доступность специалистов.** На рынке Красноярска (и РФ в целом) Elementor-разработчик — самый доступный и дешёвый специалист. Freelance: 500-1500 руб./час. Замена разработчика безболезненна.

### Минусы

1. **Тяжёлый фронтенд.** Elementor генерирует 300-500 КБ CSS + 200-400 КБ JS на каждую страницу. С JetEngine/JetSmartFilters — ещё +100-200 КБ. При 205 фото и мобильных 76.7% — PageSpeed Mobile реально получить 45-65, а не 90+. Google Core Web Vitals (INP) — проблемная зона.
2. **jQuery-зависимость.** Elementor до сих пор зависит от jQuery (~90 КБ). Это legacy-архитектура, которая замедляет время до интерактивности (TTI) на мобильных устройствах.
3. **Vendor lock-in.** Все шаблоны хранятся в проприетарном формате Elementor. Переход на другой builder = пересборка всего сайта с нуля. CSS-классы генерируются автоматически (`.elementor-element-abc123`), нет семантики.
4. **Подписочная модель.** $59/год за 1 сайт (Essential), $199/год за 25 сайтов. При отказе от подписки — виджеты Pro перестают работать, формы ломаются, динамический контент пропадает. Для квиза и каталога это критично.
5. **Конфликты обновлений.** Elementor + ACF + JetEngine + RankMath + кэш-плагин — 5+ плагинов, которые могут конфликтовать при обновлении. В практике: 2-3 раза в год что-то ломается после обновления.
6. **Дублирование DOM.** Elementor оборачивает каждый элемент в 3-5 уровней div. Карточка кухни, которая должна быть `<article><img><h3><p>`, превращается в 15+ вложенных div. Это вредит SEO (семантика) и производительности (рендеринг).

### Когда лучше

Идеален для проектов, где скорость запуска важнее производительности, контент-менеджер будет активно работать с сайтом ежедневно, бюджет на разработку минимален, и нет амбиций по SEO-росту из поиска.

## Вариант 2: WordPress + Bricks Builder

### Суть подхода

Визуальный конструктор, который работает как WordPress-тема (не плагин). Генерирует чистый семантический HTML с class-based CSS. Нет jQuery-зависимости. Одноразовая покупка (lifetime). Встроенные Query Loop, условная логика, динамический контент — без сторонних аддонов.

### Плюсы

1. **Производительность на 30-50% выше Elementor.** Bricks генерирует 50-100 КБ CSS (против 300-500 КБ у Elementor), 0 КБ jQuery. На мобильных устройствах (76.7% трафика) это разница между PageSpeed 50 и 85+. INP (Interaction to Next Paint) — в зелёной зоне из коробки.
2. **Class-based CSS система.** Стили привязаны к классам (`.kitchen-card`, `.hero-section`), а не к ID элементов. Это профессиональная архитектура: стили переиспользуются, легко поддерживаются, CSS-файл компактный. Design tokens из brand-guide.md (`--color-primary: #C2613A`, `--space-4: 16px`) нативно интегрируются.
3. **Встроенная динамика без плагинов.** Query Loop для каталога 42 кухонь, условный рендеринг (показать бейдж «Скидка» если `kitchen_sale_price` заполнено), динамические данные из ACF — всё встроено. Не нужен JetEngine ($43/год).
4. **Lifetime-лицензия.** ~$79 одноразово (вместо $59/год у Elementor). За 2 года экономия: $79 vs $118 + $86 (JetEngine) = $204. Экономия ~$125 (11 500 руб.).
5. **Лучший SEO из всех визуальных builder-ов.** Семантический HTML (`<article>`, `<section>`, `<nav>`), минимальный DOM nesting (3-4 уровня вместо 10-15 у Elementor), чистые URL-якоря, нативная поддержка Schema.org через RankMath.
6. **Тема, а не плагин.** Bricks = тема WordPress. Нет конфликта «тема + builder + дочерняя тема». Один слой вместо трёх. Меньше точек отказа при обновлениях.
7. **Responsive controls.** Настройка стилей для мобильных/планшетов/десктопов прямо в визуальном редакторе. Mobile-first подход встроен: задаёшь стили для мобильного, потом добавляешь для планшета и десктопа.

### Минусы

1. **Меньшее сообщество.** ~100K сайтов против 16M у Elementor. Меньше готовых шаблонов, меньше обучающих видео на русском, меньше ответов на форумах. Для специфических задач (квиз-калькулятор) придётся искать решение самостоятельно.
2. **Кривая обучения для контент-менеджера.** Интерфейс ближе к коду: панель стилей похожа на CSS-редактор, class-based система требует понимания каскадности. Для изменения цены кухни через ACF это не проблема, но для правки дизайна — нужно обучение (3-5 дней вместо 1-2 у Elementor).
3. **Квиз-калькулятор придётся строить вручную.** Нет готового виджета «пошаговая форма» как в Elementor Pro Forms. Варианты: кастомный HTML/JS внутри Bricks, или внешний плагин (Jetsteps, Gravity Forms Multi-Step, или самописный). Это +8-16 часов разработки.
4. **Vendor lock-in (тот же, что у любого builder).** Шаблоны Bricks нельзя перенести в Elementor или Gutenberg. Переезд = пересборка. Впрочем, class-based CSS можно частично переиспользовать, что лучше, чем ID-based у Elementor.
5. **Меньше аддонов.** Нет аналога JetSmartFilters для AJAX-фильтрации каталога из коробки. Есть BricksExtras ($49 lifetime) и Bricks Forge ($49 lifetime) — покрывают 90% потребностей, но это дополнительный расход.

### Когда лучше

Идеален для проектов, где производительность и SEO критичны, сайт будет расти (блог, новые модели), управление контентом в основном через ACF-поля (а не через визуальный редактор), и есть разработчик среднего уровня.

### Сложность реализации

### Поддержка

- **Нужен ли разработчик для контента?** Нет — контент (цены, фото, характеристики) управляется через ACF-поля в админке WordPress. Для правок дизайна — да, нужен.
- **Сложность обновлений:** Низкая. Bricks = тема, обновляется как обычная тема. Меньше зависимостей.

## Вариант 3: WordPress + Full Site Editing (Gutenberg Native Blocks)

### Суть подхода

Нативный редактор WordPress без сторонних page builder-ов. Блочная тема (block theme), Site Editor для шаблонов, паттерны блоков, Global Styles. Всё на стандартных WordPress-блоках + кастомные ACF-блоки для специфических элементов (карточка кухни, квиз, фильтры).

### Плюсы

1. **Нулевые затраты на builder.** Gutenberg — встроен в WordPress. Нет лицензий за page builder. Экономия $59-79 в первый год и далее ежегодно.
2. **Максимальная совместимость с будущим WordPress.** FSE — официальное направление развития WordPress. Все новые функции ядра (Global Styles, Query Loop block, Navigation block) работают нативно. Нет риска, что builder перестанет поддерживаться.
3. **Лёгкий фронтенд.** Нет JS-фреймворка builder-а (ни Elementor-овские 400 КБ, ни даже 100 КБ Bricks). Только стили темы + стили блоков. При правильной block-теме — PageSpeed 90+ достижим.
4. **Стандартный формат данных.** Контент хранится в стандартном HTML в `post_content` — никакого проприетарного JSON. Переезд на любую другую систему проще, чем с Elementor/Bricks.
5. **REST API из коробки.** WordPress REST API + `show_in_rest: true` (уже настроено в cpt-kitchen.json) = готовый headless backend. Если через год захотим перейти на Next.js — данные уже доступны по API.

### Минусы

1. **Незрелость FSE для сложных макетов.** По состоянию на март 2026 FSE значительно улучшился, но всё ещё уступает Elementor/Bricks по гибкости визуального редактирования. Создать нестандартный hero-баннер или сложную сетку карточек — требует больше усилий.
2. **Кастомные блоки = React-разработка.** Для карточки кухни с ACF-полями, квиз-калькулятора, AJAX-фильтрации каталога нужно писать кастомные блоки на React (или регистрировать ACF blocks). Это 20-40 часов дополнительной разработки по сравнению с визуальными builder-ами.
3. **Визуальное редактирование ограничено.** Контент-менеджер может менять текст и изображения в блоках, но не может перестраивать макет страницы без понимания системы шаблонов и паттернов. Менее интуитивно, чем drag-and-drop.
4. **Нет готовой AJAX-фильтрации.** Для каталога 42 кухонь с фильтрами по типу/стилю/материалу/цене нужен либо плагин (FacetWP, ~$99/год), либо кастомная разработка на REST API + JavaScript. Ни один нативный блок не даёт это из коробки.
5. **Квиз-калькулятор = полностью кастомный.** В экосистеме Gutenberg нет готового решения для пошагового квиза с визуальным выбором планировки. Только внешние плагины (не блочные) или полностью кастомная разработка.
6. **Экосистема шаблонов слабее.** Готовых block-тем для мебельного бизнеса практически нет. Придётся строить тему с нуля или адаптировать GenerateBlocks / Developer-oriented тему.

### Когда лучше

Идеален для блогов, корпоративных сайтов с простой структурой, проектов с долгосрочным горизонтом (5+ лет), где важна совместимость с ядром WordPress и минимальная зависимость от сторонних плагинов. Хорош как backend для headless-архитектуры.

## Вариант 4: Next.js + Headless WordPress (WPGraphQL)

### Суть подхода

WordPress используется только как backend (CMS) для управления контентом. Фронтенд — отдельное приложение на Next.js (React), которое забирает данные через WPGraphQL или REST API. Сайт генерируется статически (SSG) или рендерится на сервере (SSR). Деплой фронта — на Vercel/VPS.

### Плюсы

1. **Максимальная производительность.** SSG (Static Site Generation) — каждая страница = HTML-файл, отдаётся за 50-100 мс. PageSpeed Mobile 95-100 достижим. Для 42 карточек кухонь статическая генерация — идеальный сценарий. Core Web Vitals — всё в зелёной зоне.
2. **Полный контроль фронтенда.** React-компоненты: квиз-калькулятор с визуальным выбором планировки, AJAX-фильтрация каталога с мгновенным откликом, анимации (Framer Motion), интерактивный before/after слайдер — всё без ограничений builder-ов.
3. **Оптимальный mobile-first.** Next.js Image (автоматический WebP, srcset, lazy loading, blur placeholder), CSS-in-JS или Tailwind — pixel-perfect адаптация для 76.7% мобильного трафика. LQIP-превью из brand-guide.md реализуется нативно.
4. **Современный DX (Developer Experience).** TypeScript, компонентная архитектура, Git-based workflow, CI/CD через GitHub Actions (уже настроен). Hot reload, линтинг, тесты. Код ревьюится, версионируется, деплоится автоматически.
5. **Headless = гибкость.** WordPress-backend можно переиспользовать для мобильного приложения, Telegram Mini App, или другого фронтенда. WPGraphQL + ACF-поля кухонь (cpt-kitchen.json с `show_in_rest: true`) — данные доступны из любого клиента.
6. **ISR (Incremental Static Regeneration).** Заказчик меняет цену кухни в WordPress — через 60 секунд статическая страница автоматически обновляется. Не нужен полный ребилд.

### Минусы

1. **Сложность разработки x3-5.** Два отдельных приложения: WordPress (backend) + Next.js (frontend). Настройка WPGraphQL, типизация данных (TypeScript), маршрутизация, preview mode, SEO-рендеринг (getStaticProps/getServerSideProps) — это 200-300 часов вместо 80-110 у Bricks.
2. **Заказчик не видит результат в WordPress.** В админке WordPress — только поля ACF. Нет визуального превью «как выглядит на сайте». Preview mode работает, но через отдельный URL. Для контент-менеджера без опыта это дезориентирует.
3. **Двойной хостинг.** WordPress на VPS (или managed hosting) + Next.js на Vercel (free tier: 100 ГБ bandwidth) или том же VPS. Два процесса, два деплоя, два мониторинга. Watchdog на VPS (уже есть) придётся расширять.
4. **Over-engineering для 42 карточек.** Next.js + WPGraphQL — это архитектура для сайтов с 1000+ страниц, мультиязычностью, A/B-тестами, персонализацией. Для 8-10 страниц + 42 карточек кухонь регионального мебельного бизнеса это стрельба из пушки по воробьям.
5. **Проблемы с превью и черновиками.** WordPress draft/preview → Next.js требует настройки preview mode (отдельный API route + cookie + bypass ISR). Это 4-8 часов настройки и постоянный источник багов.
6. **Нет визуального редактора.** Макеты страниц — в React-компонентах. Чтобы изменить порядок секций на главной или добавить новый блок «Акция» — нужен React-разработчик. Через chat.mebelit.ru (Claude) можно, но это не то же, что drag-and-drop.
7. **SEO-подводные камни.** CSR (Client-Side Rendering) убивает SEO. SSR/SSG решает проблему, но требует правильной настройки. Мета-теги, Open Graph, canonical URL, structured data — всё вручную (next-seo, next-sitemap). RankMath не работает на фронтенде.

### Когда лучше

Идеален для проектов с высокой нагрузкой (10K+ страниц), мультиязычных сайтов, маркетплейсов, проектов с командой из 3+ разработчиков, и когда требуется максимальная производительность + полный контроль UX.

### Поддержка

- **Нужен ли разработчик для контента?** Для контента — нет (ACF). Для изменения дизайна, секций, страниц — да, React-разработчик обязателен.
- **Сложность обновлений:** Высокая. Два приложения, зависимости npm, совместимость WPGraphQL с WordPress, версии Next.js.
- **Доступность специалистов:** Низкая. React + WordPress headless — нишевый стек. В Красноярске: 0-3 специалиста. Удалённо: 2000-5000 руб./час.

## Рекомендация: WordPress + Bricks Builder

### Почему именно Bricks для «Кухни Рема»

**Bricks Builder — оптимальный баланс между производительностью, SEO, стоимостью и управляемостью для данного конкретного проекта.**

Вот почему:

**1. SEO с нуля требует скорости — Elementor её не даст.**
Сейчас у сайта 18 органических визитов за 90 дней. Цель — 1500+/мес за 12 месяцев. Google и Яндекс учитывают Core Web Vitals (LCP, INP, CLS) при ранжировании. На мобильных (76.7% аудитории) Elementor-сайты стабильно показывают PageSpeed 45-65 — это жёлтая/красная зона. Bricks даёт 75-90 — зелёная зона. При прочих равных условиях Bricks-сайт будет ранжироваться выше Elementor-сайта.

**2. 42 карточки кухонь + 205 фото — нужен лёгкий код.**
Каталожные страницы с 10-20 карточками (фото + цена + характеристики) — это тяжёлая страница. У Elementor каждая карточка = 15+ div-обёрток. У Bricks — 3-4. При 20 карточках на странице разница: 300 div (Elementor) vs 80 div (Bricks). На мобильном с 3G это разница в 1-2 секунды загрузки.

**3. Design tokens из brand-guide.md встраиваются нативно.**
Бренд-гайд Кухни Рема определяет CSS-переменные: `--color-primary: #C2613A`, `--space-4: 16px`, `--font-size-4xl` и т.д. Bricks с class-based CSS позволяет создать `design-tokens.css` и использовать переменные глобально. В Elementor это возможно, но через «Global Colors/Fonts» — менее гибко и не покрывает spacing/radius.

**4. Стоимость владения за 2 года — самая выгодная.**
Bricks lifetime ($79) vs Elementor подписка ($59/год x 2 = $118) + JetEngine ($43/год x 2 = $86) = $204 у Elementor vs $79 у Bricks. За 2 года Bricks экономит ~11 500 руб. на лицензиях, при этом даёт лучшую производительность.

**5. Next.js — избыточен, FSE — незрел.**
Next.js даёт идеальную производительность, но 200-300 часов разработки (vs 80-110 у Bricks) и зависимость от React-разработчика — это риск для регионального мебельного бизнеса. FSE Gutenberg хорош теоретически, но кастомные блоки для квиза и фильтрации каталога — это React-разработка по цене Next.js, без преимуществ полного контроля.

**6. Управление через ACF + chat.mebelit.ru — достаточно.**
Заказчик не будет ежедневно перестраивать макеты в визуальном редакторе. Основные операции: обновить цену кухни, добавить фото, изменить телефон, добавить модель. Всё это — ACF-поля, которые работают одинаково в любом стеке. Преимущество Elementor в drag-and-drop для этого проекта нерелевантно.

### Что Bricks даст на этапе запуска (первые 2 недели)

- Установленный WordPress + Bricks + ACF Pro + RankMath на VPS
- CPT Kitchen с 42 моделями кухонь, загруженными из VK Market (205 фото)
- Главная страница с hero-баннером, каталогом-превью, встроенным квизом
- 3 каталожные страницы с Query Loop и базовой фильтрацией по типу
- Шаблон детальной карточки кухни с галереей и характеристиками
- AJAX-формы с webhook в n8n -> Telegram + Google Sheets
- Mobile-first адаптация всех страниц
- Базовый SEO: title/meta, Schema.org LocalBusiness, sitemap, robots.txt

### Что Bricks даст через 6-12 месяцев

- PageSpeed Mobile 80+, Core Web Vitals в зелёной зоне
- Органический трафик 500-1500 визитов/мес (при ведении блога)
- AJAX-фильтрация каталога по типу/стилю/материалу/цене (BricksExtras)
- Блог с SEO-статьями (нативная поддержка Bricks)
- Портфолио реализованных проектов
- Интеграция аналитики: Яндекс Метрика + вебвизор + цели
- Стабильная работа без ежемесячной платы за builder

### Какие ограничения принимаем

1. **Квиз-калькулятор потребует больше работы**, чем в Elementor (+8-16 часов). Решение: использовать Bricks Form Steps + кастомный JS, или интегрировать внешний сервис (Marquiz, Envybox).
2. **Найти Bricks-специалиста сложнее**, чем Elementor. Решение: задачи по поддержке решаются через chat.mebelit.ru (Claude), а не через фрилансера.
3. **Контент-менеджер не сможет менять дизайн** так же легко, как в Elementor. Решение: контент = ACF-поля (цены, фото, тексты), дизайн фиксирован и меняется разработчиком.

### Почему остальные 3 варианта слабее для «Кухни Рема»

**Elementor Pro** — слишком тяжёлый. При 76.7% мобильного трафика и цели SEO-роста с нуля — PageSpeed 45-65 неприемлем. jQuery, 400 КБ CSS, глубокий DOM — это техдолг с первого дня. Подписочная модель + JetEngine = +25 000 руб./год навсегда. Простота drag-and-drop не нужна, когда контент управляется через ACF и Claude Chat.

**FSE Gutenberg** — незрел для коммерческого каталога. Кастомные блоки для квиза и фильтрации = React-разработка по цене 2000-4000 руб./час. При 120-160 часах стоимость выходит за рамки бюджета (стратегия B: 120-200 тыс.). Специалистов мало. Выигрыш по производительности минимален по сравнению с Bricks, а скорость разработки — в 2 раза ниже.

**Next.js + Headless WP** — over-engineering. Для 42 карточек кухонь и 8 страниц — это стрельба из пушки по воробьям. 200-300 часов разработки, зависимость от React-разработчика, двойной хостинг, отсутствие визуального превью для заказчика. Производительность идеальна, но Bricks даёт 80-90% этой производительности при 40% стоимости. Если через 2-3 года бизнес вырастет до федерального уровня — можно мигрировать, данные в WordPress REST API уже готовы.
